## 1. Model Card Template (AI Content Detector)

Use this as a living document, updated with every model retrain/version bump. Publish an abridged version to institutional customers; keep the full version internal.

```markdown
# Model Card: [Model Name / Version, e.g. AIDetect-v2.3]

## Model Overview
- **Purpose**: Estimate likelihood that submitted text was generated by an LLM.
- **Intended use**: Decision-support signal for human reviewers in academic
  integrity workflows. NOT intended as sole/automated grounds for disciplinary action.
- **Out-of-scope use**: Legal evidence, employment screening, journalism
  authenticity verification, or any fully-automated adverse action against an individual.
- **Model type**: [e.g., Ensemble — fine-tuned DeBERTa-v3 classifier +
  statistical perplexity/burstiness scorer]
- **Version**: v2.3
- **Release date**: YYYY-MM-DD
- **Training cutoff**: Generator models covered up to [e.g., GPT-4o, Claude
  Sonnet 4, Gemini 2.0 — list explicitly, this matters]

## Training Data
- **Human-written corpus**: source, size, language/dialect distribution,
  academic level distribution (high school / undergrad / grad), disciplines covered
- **AI-generated corpus**: which generator models, what prompting strategies
  (zero-shot essay prompts vs. paraphrase-of-human-text — these produce very
  different signal and must be tracked separately)
- **Paraphrased/"humanized" AI corpus**: explicitly included or not; if not,
  state this as a known blind spot
- **Known gaps**: languages not covered, dialects underrepresented, domains
  (STEM/technical writing, creative writing, ESL writing) with limited samples

## Performance Metrics (report per-bucket, never only aggregate)

| Subgroup                        | Precision | Recall | False Positive Rate |
|----------------------------------|-----------|--------|----------------------|
| Native English, undergrad essays | —         | —      | —                    |
| ESL writers                      | —         | —      | —                    |
| STEM/technical writing           | —         | —      | —                    |
| Short submissions (<150 words)   | —         | —      | —                    |
| AI text + light human editing    | —         | —      | —                    |
| AI text + paraphrase tool        | —         | —      | —                    |
| Human text resembling AI patterns (formulaic, non-native) | — | — | — |

## Known Limitations
- False positive rate on [subgroup] is elevated — state the number.
- Detection accuracy degrades against generator models released after
  [training cutoff date].
- Adversarial paraphrasing/"humanizer" tools reduce recall by approximately [X]%.
- Model has NOT been validated for [languages/domains not covered].

## Ethical Considerations
- Consequences of false positives (student accused of misconduct) are
  asymmetric and severe compared to false negatives — document how thresholds
  were chosen with this asymmetry in mind.
- Disparate impact analysis across the demographic/linguistic subgroups above.

## Recommended Use Threshold
- State the confidence/score threshold below which output should NOT be
  surfaced to reviewers at all (to avoid anchoring bias on noise).

## Maintenance
- Retraining cadence: [e.g., quarterly, or triggered by major new LLM release]
- Owner / contact for reporting suspected false positives
```

## 2. Data Processing Agreement (DPA) Structure

This is a skeleton for the DPA between you (processor) and an institutional customer (controller — e.g., a university). Have actual counsel draft the final version; this is a structural checklist, not legal text.

```markdown
# Data Processing Agreement — Structural Outline

1. **Parties & Roles**
   - Controller: [Institution] — determines purposes of processing
   - Processor: [Your company] — processes data on Controller's instructions

2. **Subject Matter & Duration**
   - Nature of processing: text similarity checking, AI-content detection
   - Duration: tied to underlying service contract term

3. **Categories of Data Subjects**
   - Students, instructors, (TAs if applicable)

4. **Categories of Personal Data**
   - Submitted document text/content
   - Metadata: name, student ID, email, course/institution identifiers
   - Derived data: similarity scores, AI-detection scores, embeddings/fingerprints

5. **Processing Purposes (must be exhaustive — no "and other purposes")**
   - Plagiarism similarity checking against defined corpora
   - AI-content likelihood scoring
   - Explicitly EXCLUDE any secondary use (e.g., "improving our models using
     this institution's student data") unless separately and explicitly
     consented to — this is the single most common DPA failure point

6. **Sub-processors**
   - List cloud hosting, vector DB provider, any third-party ML APIs used
   - Right for Controller to object to new sub-processors on notice

7. **Data Retention & Deletion**
   - Default retention period (explicit number, not "as needed")
   - Whether submissions are added to a comparison corpus for FUTURE
     submissions — if yes, this needs its own consent clause and a
     student-facing opt-out mechanism, separate from institutional consent
   - Deletion process on contract termination, with certification of deletion

8. **Cross-Border Transfer**
   - Where data is hosted/processed geographically
   - Transfer mechanism if EU data leaves EEA (SCCs, adequacy decision, etc.)

9. **Security Measures**
   - Encryption at rest/in transit, access controls, breach notification
     timeline (commonly 72 hours to match GDPR)

10. **Data Subject Rights Support**
    - Process for Controller to relay access/erasure/correction requests
    - Your SLA for fulfilling these

11. **Audit Rights**
    - Controller's right to audit or request compliance evidence

12. **Liability & Indemnification**
    - Allocation of liability for breach, misuse, or false-positive-driven
      disciplinary harm claims (this is a newer, product-specific risk class
      worth flagging explicitly rather than relying on generic data-breach clauses)

13. **Region-Specific Riders**
    - FERPA rider (US education records)
    - GDPR Article 28 processor obligations (EU)
    - State-level student privacy laws where applicable (e.g., US state
      student data privacy statutes)
```

**Two things to resolve before this goes to counsel:**
- Decide now whether the "future comparison corpus" retention model
  (Turnitin's approach) is part of your product. It's the single biggest
  legal-complexity multiplier in the whole document — cutting it simplifies
  the DPA substantially.
- Decide your stance on training your *own* detection models on customer
  submissions. If yes, this needs its own explicit, separately-flagged
  consent clause — bundling it into general "processing purposes" is a
  common source of institutional pushback and regulatory risk.

## 3. Classifier Training Pipeline (AI-Content Detector)

```
┌─────────────────────────────────────────────────────────────┐
│ 1. Data Collection                                           │
├─────────────────────────────────────────────────────────────┤
│ Human corpus:                                                │
│   - Licensed academic writing datasets                       │
│   - Historical (pre-LLM, pre-2020) student submissions       │
│     → guarantees zero AI contamination                       │
│   - Diverse: native/ESL, multiple disciplines, multiple      │
│     academic levels                                          │
│                                                                │
│ AI corpus:                                                    │
│   - Generate with each target LLM family, multiple prompt    │
│     strategies:                                               │
│       a) Direct essay generation from assignment prompt      │
│       b) AI paraphrase of an existing human essay            │
│       c) AI-generated then run through a "humanizer" tool    │
│       d) Human-written + AI-polished (mixed authorship)      │
│   - Label each with generator model + strategy — do NOT      │
│     collapse into one "AI" bucket, since these have very     │
│     different detection difficulty                           │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 2. Preprocessing                                              │
├─────────────────────────────────────────────────────────────┤
│  - Normalize formatting, strip metadata                       │
│  - Segment into sentence + paragraph + full-doc granularity   │
│    (train/eval at all three levels — sentence-level detection │
│    is needed for highlighting, doc-level for the summary score)│
│  - Stratified train/val/test split BY SOURCE DOCUMENT, not    │
│    by sentence (prevents leakage — sentences from the same    │
│    essay must stay in the same split)                         │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 3. Feature Extraction (for ensemble's statistical branch)     │
├─────────────────────────────────────────────────────────────┤
│  - Perplexity under a reference LM                            │
│  - Burstiness (variance in sentence-level perplexity)         │
│  - Lexical diversity, POS-tag distribution                    │
│  - Sentence-length variance                                   │
│  - n-gram frequency deviation from human baseline              │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 4. Model Training — Ensemble Branches                         │
├─────────────────────────────────────────────────────────────┤
│  Branch A: Fine-tuned transformer classifier                  │
│    - Base: DeBERTa-v3 or RoBERTa-large                        │
│    - Binary or multi-class head (human / AI-direct /          │
│      AI-paraphrased / AI-humanized / mixed)                   │
│    - Class-balanced loss (AI-paraphrased class is usually      │
│      underrepresented and hardest — don't let it get drowned) │
│                                                                 │
│  Branch B: Statistical/feature-based classifier                │
│    - Gradient-boosted tree (XGBoost/LightGBM) on the           │
│      engineered features from step 3                          │
│    - More interpretable, useful for explaining flags to        │
│      reviewers ("high burstiness deviation" vs. black-box)     │
│                                                                 │
│  Combine: weighted ensemble or meta-learner (logistic          │
│  regression over both branches' outputs) → final score +       │
│  confidence interval from inter-branch agreement                │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 5. Adversarial Robustness Testing                              │
├─────────────────────────────────────────────────────────────┤
│  - Run held-out test set through common paraphrase/humanizer   │
│    tools NOT seen during training — measure recall degradation │
│  - Synonym-substitution attacks, sentence-reordering attacks   │
│  - Report robustness numbers in the model card, don't bury them│
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 6. Fairness / Subgroup Evaluation                              │
├─────────────────────────────────────────────────────────────┤
│  - Compute precision/recall/FPR per subgroup (native vs ESL,   │
│    per discipline, per academic level) — this table feeds      │
│    directly into the model card                                │
│  - If FPR gap between subgroups exceeds your defined threshold,│
│    block release and go back to data collection (usually the   │
│    fix is more diverse human training data, not a bigger model)│
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 7. Calibration                                                  │
├─────────────────────────────────────────────────────────────┤
│  - Temperature scaling / isotonic regression so that a "70%"   │
│    output actually corresponds to ~70% empirical AI rate on    │
│    held-out data — raw model logits are usually miscalibrated  │
│  - Derive the confidence-band ranges (see prior response)      │
│    from calibrated inter-branch + inter-sample variance        │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 8. Deployment & Monitoring                                      │
├─────────────────────────────────────────────────────────────┤
│  - Shadow-mode deployment first: run in parallel with existing  │
│    process, don't surface to end users, compare outputs         │
│  - Drift monitoring: track score distribution over time —       │
│    a shift usually signals a new LLM generation the model       │
│    hasn't seen                                                  │
│  - Feedback loop: log human-reviewer overrides/appeals as new   │
│    labeled data for the next retrain cycle                      │
│  - Retrain trigger: scheduled (quarterly) OR event-driven       │
│    (major new LLM release, e.g., a new GPT/Claude/Gemini        │
│    generation)                                                  │
└─────────────────────────────────────────────────────────────┘
```

**Critical pipeline notes:**
- The train/val/test split by source document (not by sentence) is the
  single most common leakage bug in this kind of system — worth double-checking.
- Treat "AI-generated then paraphrased/humanized" as its own labeled class
  throughout the pipeline, not folded into generic "AI." It's the hardest
  case and the one where false negatives cluster.
- Fairness evaluation is a release gate, not a nice-to-have — given the
  severity of false-positive consequences (academic misconduct accusations),
  a subgroup FPR gap is a shippability blocker, not a footnote.
